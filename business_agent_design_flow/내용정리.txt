문제/현상
사내 AGENT BUILDER 환경은 갖춰져 있지만, 일반 사용자는 자신의 업무를 어떤 단위로 쪼개고 어디에 AI Agent를 붙여야 하는지 판단하기 어렵다. 업무 자동화 아이디어가 있어도 현재 업무 절차, 사용 데이터, 시스템, 승인/검토 지점, 기대 산출물을 Agent 설계로 바꾸는 과정이 숙련자 의존적이다.
AS-IS에서는 사용자가 “내 업무를 자동화하고 싶다”는 수준의 문제의식은 있어도, 어떤 기능을 써야 하는지, 기존 개선 사례와 어떻게 연결해야 하는지, 안전하게 자동화할 수 있는 범위가 어디까지인지 알기 어렵다. 그 결과 Agent Builder 활용이 일부 숙련자 중심으로 제한되고, 반복 업무 개선 기회가 발굴되지 못하거나 PoC까지 가는 시간이 길어진다.
개선/제안 사항
business_agent_design_flow는 사용자가 업무 설명 한 칸만 입력하면, 현재 업무 Flow와 AI Agent 적용 후 개선 Flow를 자동으로 설계해주는 안내형 Flow로 구현한다.
TO-BE는 다음 구조다. 사용자의 자연어 업무 설명을 먼저 업무 목적, 현재 단계, 데이터/시스템, 제약사항으로 구조화한다. 이후 MongoDB에 등록된 기능 목록과 기존 개선 사례를 검색해 관련 기능을 추천하고, AI Agent 설계 결과를 JSON으로 검증한 뒤 HTML 형태의 Before/After 업무 Flow로 보여준다.
주요 제안 방향은 다음과 같다.
단일 입력 UX: 사용자는 복잡한 설정값을 넣지 않고 자신의 업무를 문장으로 설명한다.
AS-IS/TO-BE 자동 도출: 현재 사람이 수행하는 단계와 AI Agent 적용 후 바뀌는 단계를 비교해 보여준다.
기능/사례 기반 추천: Agent Builder에서 활용 가능한 기능, 기존 Flow, 개선 사례를 카탈로그 기반으로 연결한다.
구현 가이드 제공: 초보자도 따라 할 수 있도록 필요한 Langflow 노드, 입력/출력, 연결 방식, 검증 기준을 제시한다.
리스크 통제: 메일 발송, 시스템 변경, 승인, 개인정보 처리처럼 위험한 작업은 사람 검토 지점을 명확히 둔다.
공유 가능한 결과물: HTML 결과와 다운로드 링크를 제공해 회의, 검토, PoC 기획 자료로 바로 활용할 수 있게 한다.
기대효과
이 Flow를 통해 Agent Builder를 처음 쓰는 사용자도 자신의 업무를 AI 기반 개선 과제로 구체화할 수 있다. 막연한 자동화 니즈가 “현재 업무 단계, 개선 후 단계, 적용 기능, 구현 방법, 검증 기준”으로 정리되기 때문에 PoC 착수 장벽이 낮아진다.
또한 개선 아이디어가 개인 경험에만 의존하지 않고 사내 기능/사례 카탈로그와 연결되므로, 이미 만들어진 Flow와 패턴을 재사용할 수 있다. 반복 업무 발굴, 자동화 후보 선정, 구현 방향 수립 시간이 줄고, 부서별로 흩어져 있던 Agent 활용 경험을 축적하는 효과도 기대할 수 있다.
결과적으로 이 Flow는 단순한 설계 보조 도구가 아니라, 사내 구성원이 자신의 업무를 Agent Builder 관점으로 재해석하도록 돕는 진입점 역할을 한다. 업무 담당자는 개선 가능성을 빠르게 확인하고, 개발자나 운영자는 표준화된 설계 결과를 바탕으로 더 안전하고 일관된 AI Agent 구현을 이어갈 수 있다.
