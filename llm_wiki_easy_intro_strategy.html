AI를 잘 모르는 사람도 이해할 수 있는 설명

LLM Wiki는 “말로 풀어쓴 업무 이야기”를 회사가 쓸 수 있는 지식으로 정리해주는 AI 편집실입니다.

직원이 복잡한 양식을 채울 필요는 없습니다. 자기 업무를 평소 말하듯이 설명하면 됩니다. LLM Wiki는 그 설명을 읽고 업무 절차, 필요한 데이터, 계산 로직, 산출물, 담당자, 확인이 필요한 부분으로 나누어 정리합니다. 회사 입장에서는 흩어진 업무 경험과 데이터 사용법을 하나씩 공식 지식으로 바꾸는 방식입니다.

핵심은 “사람이 형식에 맞춰 쓰는 것”이 아니라, “사람은 자연스럽게 쓰고 AI가 회사에 필요한 형식으로 정리하는 것”입니다.

1. 아주 쉽게 말하면: LLM Wiki는 회사 지식을 정리해주는 AI 사서입니다

LLM Wiki를 한 번에 이해하려면 단어를 나눠보면 됩니다. LLM은 글을 읽고, 요약하고, 분류하고, 질문할 수 있는 AI입니다. Wiki는 여러 사람이 함께 찾아볼 수 있도록 지식을 모아둔 공책입니다. 그래서 LLM Wiki는 AI가 회사의 여러 자료를 읽고, 사람이 찾아보기 쉬운 공책처럼 정리해주는 방식이라고 볼 수 있습니다.

LLM은 읽고 이해하는 AI

사람이 쓴 업무 설명, 회의록, 보고서, SQL, 엑셀 내용을 읽고 “무슨 이야기인지” 파악합니다. 완벽한 선생님은 아니지만, 글을 읽고 정리하는 조교처럼 일할 수 있습니다.

Wiki는 함께 보는 정리 공책

개인 노트가 아니라 모두가 찾아볼 수 있는 정리 공간입니다. 업무 절차, 데이터 위치, 계산 기준, 담당자, 자주 묻는 질문이 한곳에 모입니다.

LLM Wiki는 AI가 돕는 회사 백과사전

AI가 초안을 만들고, 사람이 맞는지 확인합니다. 승인된 내용만 공식 지식이 되므로 “AI가 마음대로 만든 문서”가 아니라 “AI가 정리하고 사람이 확인한 문서”입니다.

비유 1. 엉망인 책가방을 과목별 파일철로 정리하는 일

회사 안의 업무 자료는 책가방 안에 아무렇게나 들어간 종이와 비슷합니다. 회의록은 여기, 엑셀은 저기, 개인 메모는 다른 폴더에 있습니다. LLM Wiki는 그 종이들을 꺼내서 “이건 수학 공책”, “이건 숙제”, “이건 시험 범위”처럼 나누어 파일철에 꽂아줍니다. 회사식으로 말하면 “이건 업무 절차”, “이건 데이터 출처”, “이건 계산 로직”, “이건 확인 필요 사항”으로 나누어 정리하는 것입니다.

비유 2. 요리 레시피를 만드는 일

어떤 사람이 맛있는 음식을 만들 줄 알아도 레시피가 없으면 다른 사람은 따라 하기 어렵습니다. 회사 업무도 같습니다. 담당자는 매월 보고서를 만들 수 있지만, 다른 사람이 하려면 어떤 재료를 쓰는지, 어디서 꺼내는지, 어떤 순서로 섞는지 알아야 합니다. 여기서 재료는 데이터, 재료 보관 장소는 시스템/테이블, 조리법은 계산 로직, 완성 음식은 보고서나 대시보드입니다. LLM Wiki는 담당자가 말로 설명한 업무를 이런 레시피 형태로 바꿔줍니다.

비유 3. 회사 안의 길찾기 지도를 만드는 일

처음 가는 건물에서 “3층 왼쪽 끝 방, 두 번째 서랍에 있어요”라고 알려주면 찾기 쉽습니다. 데이터도 마찬가지입니다. “매출 데이터는 ERP에 있어요”만으로는 부족합니다. 어느 시스템, 어느 DB, 어느 테이블, 어느 컬럼인지 알아야 합니다. 또 그 값을 어떤 기준으로 계산해서 어느 보고서에 쓰는지도 알아야 합니다. LLM Wiki는 이런 길찾기 지도를 만들어 “이 숫자가 어디에서 와서 어디로 가는지”를 따라갈 수 있게 해줍니다.

일반 챗봇과 LLM Wiki의 차이

구분 일반 챗봇 LLM Wiki
역할 질문을 받으면 그때그때 답변합니다. 답변의 근거가 되는 업무 설명서와 데이터 설명서를 계속 쌓습니다.
비유 물어보면 대답하는 안내 직원에 가깝습니다. 책을 읽고 분류하고 목차를 만드는 사서에 가깝습니다.
회사에 남는 것 대화 기록은 남을 수 있지만, 공식 지식이 자동으로 정리되지는 않습니다. 업무 절차, 데이터 위치, 계산 로직, 담당자, 확인 필요 사항이 지식으로 남습니다.
중요한 점 답변이 맞는지 매번 확인하기 어렵습니다. AI가 정리한 초안을 사람이 승인해야 공식 지식이 됩니다.

LLM Wiki 안에는 4개의 방이 있다고 생각하면 쉽습니다

  • 원본방: 직원이 올린 업무 설명, 엑셀, 회의록, SQL, 보고서가 그대로 보관되는 곳입니다.
  • 정리방: AI가 원본을 읽고 업무 절차, 데이터 위치, 계산 로직으로 나누는 곳입니다.
  • 확인방: 담당자가 “이 내용이 맞다/틀리다/확인 필요”를 표시하는 곳입니다.
  • 질문방: 직원들이 승인된 지식을 바탕으로 “이 데이터 어디 있어요?” 같은 질문을 하는 곳입니다.

2. 왜 우리 회사에 필요한가?

현재 문제 1: 업무가 사람마다 흩어져 있음

어떤 사람이 어떤 일을 어떤 순서로 하는지, 어떤 파일을 보고 어떤 기준으로 판단하는지가 개인의 기억이나 개인 폴더에만 남아 있습니다. 그래서 담당자가 바뀌면 업무가 흔들립니다.

현재 문제 2: 데이터 사용법이 정리되어 있지 않음

같은 “매출”이라는 말도 어느 테이블에서 가져오는지, 주문일 기준인지 출하일 기준인지, 내부 거래를 빼는지 넣는지에 따라 값이 달라집니다. 그런데 그 기준이 공식적으로 정리되어 있지 않습니다.

그래서 목표는 챗봇이 아니라 “회사 업무와 데이터의 설명서”입니다

LLM Wiki를 만들면 직원이 질문했을 때 답변만 해주는 것이 아니라, 그 답변의 근거가 되는 업무 설명서, 데이터 설명서, 지표 정의서, 계산 로직, 사용처까지 같이 쌓이게 됩니다.

3. 직원은 그냥 서술하듯이 쓰면 됩니다

이 프로젝트가 성공하려면 직원에게 복잡한 양식을 요구하면 안 됩니다. 처음 입력 화면은 “본인의 업무를 편하게 설명해주세요” 정도여야 합니다. AI가 뒤에서 필요한 항목으로 나눕니다.

사람이 쓰는 방식 
저는 매월 초에 영업 실적 보고서를 만듭니다.
ERP에서 전월 매출 데이터를 뽑고, 내부 거래처는 제외합니다.
그 다음 엑셀 피벗으로 거래처별, 제품군별 금액을 정리해서 팀장님께 공유합니다.
가끔 ERP 값이 BI 대시보드랑 안 맞을 때가 있는데,
그럴 때는 출하일 기준인지 세금계산서 기준인지 확인합니다.
AI가 정리하는 방식 
업무명: 월간 영업 실적 보고
주기: 매월 초
입력 데이터: ERP 전월 매출 데이터
주요 로직: 내부 거래처 제외, 거래처별/제품군별 집계
산출물: 영업 실적 보고서
공유 대상: 팀장
확인 필요: ERP 원천 테이블명, 내부 거래처 제외 기준
데이터 이슈: ERP와 BI 값 불일치 가능
추가 질문: 매출 기준일은 출하일인가, 세금계산서 발행일인가?

중요한 원칙

AI가 모르는 내용을 마음대로 지어내면 안 됩니다. 모르는 것은 확인 필요로 남겨야 합니다. 예를 들어 담당자가 테이블명을 모르면 AI는 그럴듯한 테이블명을 만들지 않고, “원천 테이블 확인 필요”라고 표시해야 합니다.

4. Data Governance도 어렵게 시작할 필요가 없습니다

Data Governance라는 말이 어렵게 들리지만, 쉽게 말하면 “회사에서 쓰는 숫자의 족보를 정리하는 일”입니다. 어떤 숫자가 어디서 왔고, 어떻게 계산됐고, 누가 책임지고, 어디에 쓰이는지를 알 수 있게 만드는 것입니다.

쉬운 질문 Data Governance에서 정리할 내용 예시
이 데이터는 어디에 있어요? 시스템, DB, 테이블, 컬럼, 파일 위치 ERP 매출 테이블, BI 모델, 월간 엑셀 파일
이 숫자는 어떻게 계산해요? 계산식, SQL, 필터 조건, 기준일, 제외 조건 전월 매출 = 출하 완료 금액 - 반품 금액
누가 이 정의를 책임져요? 업무 owner, data owner, 검수자 영업기획팀 owner, 데이터팀 steward
이 값은 어디에 써요? 보고서, 대시보드, 회의, 후속 업무 월간 경영회의, 영업 실적 대시보드
값이 다르면 어떻게 해요? 검증 기준, 예외 처리, 확인 절차 ERP와 BI 값 불일치 시 기준일과 제외 조건 확인

5. LLM Wiki와 Data Governance가 만나는 지점

1

업무 설명

담당자가 본인 업무를 말하듯이 적습니다.

2

AI 정리

AI가 업무, 데이터, 로직, 산출물로 나눕니다.

3

추가 질문

테이블명, 기준일, 제외 조건처럼 빠진 것을 물어봅니다.

4

사람 검수

담당자와 데이터 owner가 맞는지 확인합니다.

5

공식 지식화

승인된 내용만 사내 Wiki와 데이터 정의로 반영합니다.

이렇게 되면 가능한 질문

  • “월간 영업 실적 보고서는 누가 만들고 어떤 데이터로 만드나요?”
  • “매출 지표는 어느 테이블에서 가져오고 계산 기준일은 무엇인가요?”
  • “ERP와 BI 값이 다를 때 어떤 순서로 확인해야 하나요?”
  • “이 테이블을 바꾸면 어떤 보고서가 영향을 받나요?”
  • “신규 담당자가 이 업무를 인수인계받으려면 무엇을 보면 되나요?”

6. 화면은 단순해야 합니다

1단계: 자유 입력

“본인이 하는 업무를 편하게 설명해주세요.” 사용자는 이 한 칸에 글을 쓰거나 파일을 올립니다.

2단계: AI 정리 결과

AI가 업무명, 주기, 입력 데이터, 계산 로직, 산출물, 확인 필요 사항으로 나눠 보여줍니다.

3단계: 확인만 하기

사용자는 “맞음”, “수정”, “모름”, “담당자에게 확인”처럼 간단히 피드백합니다.

사용자에게 복잡한 용어를 보이지 않는 것이 중요합니다

사용자는 Data Lineage, Metadata, Steward 같은 말을 몰라도 됩니다. 대신 화면에는 “이 숫자는 어디서 가져오나요?”, “이 계산에서 제외하는 조건이 있나요?”처럼 쉬운 질문으로 보여줘야 합니다.

7. 공식화는 단계가 필요합니다

자유롭게 입력한 내용이 바로 회사 공식 지식이 되면 위험합니다. 개인 기억이 틀릴 수도 있고, 오래된 기준일 수도 있고, 보안상 공유하면 안 되는 내용이 섞일 수도 있습니다. 그래서 상태를 나눠야 합니다.

상태 뜻 사용 범위
개인 초안 담당자가 자유롭게 쓴 내용과 AI 정리 초안 본인 또는 제한된 검토자
팀 검토중 팀 업무로 쓸 수 있는지 확인하는 단계 팀 내부
팀 승인 팀에서 맞다고 인정한 업무/데이터 지식 팀 또는 관련 조직
전사 공식 여러 조직이 공통으로 사용해도 되는 공식 정의 권한이 허용된 전사 사용자

8. 처음 시작은 작게 하는 것이 좋습니다

  1. 대표 업무 하나를 고릅니다. 예를 들면 월간 실적 보고, 재고 현황 보고, 생산 지표 확인처럼 데이터와 업무 절차가 함께 있는 일이 좋습니다.
  2. 담당자에게 자유 서술을 받습니다. 처음부터 양식을 강요하지 말고, “평소 어떻게 처리하는지 말하듯이 적어주세요”라고 요청합니다.
  3. AI가 정리한 초안을 보여줍니다. 업무 절차, 데이터 위치, 계산 기준, 산출물, 확인 필요 사항으로 나눠 보여줍니다.
  4. 빠진 데이터 정보를 추가 질문합니다. 테이블명, 컬럼명, 기준일, 제외 조건, 사용 리포트 등 꼭 필요한 부분만 묻습니다.
  5. 검수된 내용만 공식 지식으로 등록합니다. 이 과정을 몇 개 업무에 반복하면 회사의 업무 Wiki와 데이터 거버넌스가 같이 쌓입니다.

9. 결론

우리 회사에 맞는 LLM Wiki 정의

LLM Wiki는 직원에게 어려운 문서 양식을 쓰게 만드는 도구가 아닙니다. 직원은 자기 업무를 자연스럽게 설명하고, AI가 그것을 업무 절차, 데이터 위치, 계산 로직, 산출물, 확인 필요 사항으로 정리해주는 도구입니다.

이 과정을 통해 회사는 “누가 어떤 업무를 어떻게 하는지”와 “어떤 데이터가 어디에서 와서 어떤 계산을 거쳐 어디에 쓰이는지”를 함께 정리할 수 있습니다. 그래서 LLM Wiki는 단순한 AI 검색 도구가 아니라, 업무 표준화와 Data Governance를 쉽게 시작하게 해주는 입구라고 보는 것이 맞습니다.

10. 실제 입력 Tool은 어떻게 만들면 좋을까?

사람들이 입력하게 만들려면 “문서 작성 시스템”처럼 보이면 안 됩니다. 가장 좋은 입력 도구는 업무를 설명하는 메모장처럼 보여야 합니다. 사용자는 편하게 쓰고, AI가 뒤에서 업무/데이터/로직/산출물로 정리하는 방식이 적합합니다.

추천 1. 업무 설명 메모장

첫 화면은 큰 입력창 하나면 충분합니다. “평소 업무를 하듯이 설명해주세요”라고 안내하고, 파일이나 링크를 함께 첨부하게 합니다.

추천 2. AI가 되묻는 인터뷰

사용자가 글을 쓰면 AI가 부족한 부분만 3~5개 질문합니다. 예를 들어 “어느 시스템에서 데이터를 받나요?”, “제외 조건이 있나요?”처럼 묻습니다.

추천 3. 확인만 하는 리뷰 화면

AI가 정리한 결과를 카드 형태로 보여주고, 사용자는 “맞음”, “수정”, “모름”, “담당자 확인 필요”만 선택하게 합니다.

입력 Tool 후보

방안 장점 한계 추천 용도
간단한 설문/폼 가장 빨리 시작 가능. 링크만 뿌리면 직원들이 바로 입력 가능 대화형 추가 질문, 검수 화면, 데이터 lineage 관리에는 약함 PoC 초반, 업무 서술 자료를 빠르게 모을 때
Teams/메신저 챗봇 직원들이 이미 쓰는 업무 공간 안에서 자연스럽게 입력 가능 정리 결과를 한눈에 검수하거나 표준 승인 흐름을 만들려면 별도 화면이 필요 업무 인터뷰, 추가 질문, 짧은 입력 수집
Low-code 입력 앱 입력, 첨부, 검수, 승인, 상태 관리까지 비교적 빠르게 구현 가능 복잡한 AI 처리나 데이터 카탈로그 연동은 별도 개발이 필요할 수 있음 1차 사내 서비스, 팀 단위 운영
전용 사내 웹앱 자유 입력, 파일 업로드, AI 정리, 검수, 승인, 데이터 연동을 원하는 방식으로 구현 가능 개발/운영 비용이 가장 큼 전사 확대, Data Governance까지 본격 운영할 때

현실적인 추천 순서

  1. 처음 2~4주는 간단한 입력 폼으로 시작합니다. 질문은 많이 넣지 말고, “업무를 자유롭게 설명해주세요”, “자주 보는 파일/시스템을 알려주세요” 정도로 둡니다.
  2. AI가 정리한 결과를 담당자에게 다시 보여줍니다. 사용자가 처음부터 구조화하지 않게 하고, AI가 만든 초안을 확인하게 해야 부담이 낮습니다.
  3. 반응이 좋으면 챗봇형 인터뷰 또는 Low-code 앱으로 확장합니다. 이때부터 추가 질문, 파일 첨부, 승인 상태, 담당자 지정, 데이터 owner 검수 기능을 붙입니다.
  4. 전사 운영 단계에서만 전용 웹앱을 고려합니다. 처음부터 큰 시스템을 만들기보다, 사람들이 실제로 어떤 식으로 입력하는지 확인한 뒤 설계하는 것이 안전합니다.

결론: 입력 Tool의 UX 원칙

사용자가 보는 화면은 최대한 말랑해야 합니다. “업무를 말하듯이 쓰세요”가 앞에 있고, “데이터 거버넌스 구조”는 뒤에 숨어 있어야 합니다. 직원이 구조를 맞추는 것이 아니라, AI가 구조를 맞추고 사람은 확인만 하는 방식이어야 입력률이 올라갑니다.
